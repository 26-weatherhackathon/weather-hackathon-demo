# 대표 데모 기획서: 골든타임 — 레이더로 대피 시간을 버는 게임

> 기상·기후 AI 해커톤 2026 운영진 대표 데모 · 기획 초안 v0.1
> 2026-07-12 팀 회의 결정 사항을 반영한 신규 방향 제안 문서

# 1. 배경과 결정 사항

## 1.1 회의 합의 (2026-07-12)

1. 주제는 교육과정에서 가져오지 않는다. 기상청의 존재 목적 중 가장 큰 비중인 "재난 대응 및 안전"에서 가져온다.
2. 다만 콘텐츠의 구성(타겟 정의, 학습 목표, 평가 기준)은 교육과정 방법론으로 구체화한다.
3. 형식은 게임 기반. 공교육이 아니므로 참여율을 높이는 것이 최우선이고, 게임이 참여율에 가장 유리하다.

## 1.2 대표 데모의 역할

이 데모는 "이 주제로 만들라"는 정답 제시가 아니라, 교육 콘텐츠를 만들 때 고려해야 할 것들(타겟 선언, 학습 목표, 실데이터 사용, 채점 설계)을 실물로 보여주는 방법론 시연이다. 본선팀에게 전달할 메시지는 다음 세 가지다.

1. 교육 대상을 스스로 좁혀 선언하라
2. 기상 데이터를 증빙 가능하게 사용하라
3. 학습자가 직접 조작하고 즉시 피드백을 받게 하라

## 1.3 직전 데모(온실효과)에서 얻은 교훈

1. 실제 기상 데이터를 쓰지 않으면 공고의 핵심 요건("실제 기상 데이터 또는 파생 모델을 직접 조작")을 놓친다. 이번에는 실데이터를 정면으로 사용한다.
2. 원리 이해에서 끝나지 않고, 배운 것이 현실 행동으로 이어지는 마무리를 넣는다.

# 2. 한 줄 정의와 타겟 선언

## 2.1 한 줄 정의

"재난문자를 기다리는 사람에서, 레이더를 읽는 사람으로."

머리 위 하늘은 맑아도 데이터는 위험을 먼저 보여준다. 레이더를 읽을 줄 알면 대피 시간을 벌 수 있고, 이 게임의 점수가 곧 그 시간이다.

## 2.2 타겟 선언 (본선팀이 베껴 쓰기를 바라는 3줄 양식)

| 항목 | 내용 |
|---|---|
| 누가 | 일반 시민 (기상 전공지식 0, 중학생 이상 플레이 가능) |
| 무엇을 | 레이더 영상과 기상특보를 읽고 위험을 남보다 먼저 판단해 대피 시간을 버는 방법 |
| 어디서 | 웹 브라우저 (모바일 대응), 1회 플레이 5~7분 |

## 2.3 교과 연계 (각주 수준으로 유지)

주 타겟은 일반 시민이지만, 아래 두 줄로 학교 현장 활용 근거를 확보한다.

1. 중학교 과학: 2022 개정 [9과17-04] 탐구활동 "일기도, 레이더 및 위성 영상 등 실시간 데이터를 활용하여 날씨 해석하기" (2027학년도 중3부터 적용, 2026학년도까지는 2015 개정 [9과18-04]에 해당)
2. 학교 안전교육 7대 표준안 중 재난안전(자연재난 대피) 영역

# 3. 학습 목표와 평가 (백워드 설계)

## 3.1 학습 목표

1. 레이더 영상에서 강수 에코의 강도(색 범례)와 이동 방향을 읽고 위험을 판단할 수 있다
2. 호우주의보와 호우경보의 차이, 특보 발표 기준을 안다
3. "내 위치의 하늘"과 "데이터가 보여주는 위험"이 다를 수 있음을 체험으로 안다
4. 플레이 후 현실 행동 1개로 연결한다 (내 지역 레이더 확인, 재난문자 수신 설정, 대피 경로 확인)

## 3.2 평가 설계

| 측정 대상 | 방법 |
|---|---|
| 판독 정확도 | 레이더 체크 시 위험도 판정 퀴즈의 정오답 (즉시 피드백) |
| 판단 타이밍 | 골든타임 지수 = 급류 도달 시각 - 대피 완료 시각 (점수의 본체) |
| 자기 언어 설명 | 설득 미션: 동행인에게 대피 근거를 고르는 선택지 (오개념 선택지 포함) |
| 행동 연결 | 엔딩의 우리 동네 모드 체크리스트 완료율 |

# 4. 게임 설계

## 4.1 장르와 레퍼런스: 타임 매니지먼트 아케이드 (쿠킹 매드니스 방식)

서사형 어드벤처가 아니라, 짧고 빠른 스테이지에서 시간 압박 아래 여러 태스크를 동시에 처리하는 타임 매니지먼트 게임으로 만든다. 쿠킹 매드니스의 "밀려오는 주문을 쳐내는 주방" 구조를 "다가오는 급류 앞의 캠핑장"으로 치환한다.

| 쿠킹 매드니스 | 골든타임 |
|---|---|
| 밀려오는 주문 | 처리해야 할 대피 태스크 (텐트 걷기, 귀중품 챙기기, 일행 깨우기, 차 이동) |
| 조리 타이머 | 급류 도달 카운트다운 (레이더 데이터가 결정) |
| 태우면 실패 | 레이더를 확인하지 않으면 상황 악화를 놓침 |
| 콤보 | 올바른 우선순위로 연속 처리 시 콤보 보너스 |
| 별 3개 클리어 | 번 시간(골든타임)에 따라 별 1~3개 |
| 다음 레벨 | 다른 재난 유형, 더 짧은 골든타임 |

## 4.2 핵심 루프 (1스테이지 2~3분)

1. 브리핑 (15초): 상황 제시. "캠핑장, 하늘은 맑음." 레이더 앱을 여는 법을 배운다
2. 플레이: 화면에 대피 태스크가 쌓인다. 태스크를 처리하면서 주기적으로 레이더를 열어 에코의 이동을 확인해야 한다
   - 레이더를 열면 실제 레이더 시계열 프레임이 갱신되고, 남은 시간이 재계산된다
   - 에코가 보라색으로 강해지면 카운트다운이 빨라진다. 레이더를 안 보면 이 변화를 모른 채 태스크만 처리하게 된다
   - 모든 행동은 시간을 소모한다. 무엇을 버리고 무엇을 챙길지가 곧 게임
3. 판정 (10초): 대피 완료 시각과 급류 도달 시각을 나란히 공개. 번 시간이 점수
4. 리플레이 유도: "레이더를 한 번 더 봤다면 몇 분을 더 벌었을까"를 보여주고 재도전 버튼

## 4.3 빠른 피드백 원칙

1. 모든 탭/클릭은 0.2초 안에 반응한다 (점수 팝업, 시간 증감 표시)
2. 판독 퀴즈의 정오답은 즉시 표시하고, 오답이어도 게임은 계속된다 (시간 손실로 반영)
3. 스테이지가 짧으므로 실패해도 곧바로 재도전할 수 있다

## 4.4 스테이지 구성 (1개 완전체 + 확장 슬롯)

| 스테이지 | 상황 | 상태 |
|---|---|---|
| 1. 계곡 캠핑장 | 상류 게릴라성 호우, 급류. 내 위치는 맑음 | 완전체로 구현 |
| 2. 도심 지하주차장 | 시간당 집중호우, 차량 이동 판단 | 잠금 슬롯 (확장 예시) |
| 3. 태풍 전야 | 특보 단계별 행동, 경보와 주의보 차이 | 잠금 슬롯 (확장 예시) |

스테이지 1만 실데이터로 완성하고, 나머지는 잠금 카드로 두어 동일 구조의 확장 가능성을 증명한다 (forecaster 계획의 1케이스 완전체 원칙 유지).

## 4.5 연출 원칙 (실제 피해 사건 존중)

1. 실패 연출은 침수/피해 장면 재현이 아니라 "골든타임을 놓쳤다"는 시계 연출로 표현한다
2. 실제 사망 사고를 시나리오로 재연하지 않는다. 상황 유형(계곡, 지하공간)만 차용한다
3. 캐릭터 사망 묘사 없음. 실패 시 "다시 그 아침으로" 되돌아가는 타임루프 프레임

# 5. 실데이터 사용 계획

## 5.1 원칙

판단 근거가 되는 데이터는 100% 실측을 사용한다. 개발 시 1회 수집하여 정적 파일화하고, 원본 응답을 raw 디렉토리에 보존하여 증빙한다. 모든 데이터 패널에 출처 라벨과 관측 시각을 표기한다. (forecaster-build-spec의 수집/증빙 패턴 재사용)

## 5.2 데이터 소스

| 데이터 | 소스 | 게임 내 용도 |
|---|---|---|
| 레이더 합성영상 시계열 | 기상청 API허브 HSR | 핵심 게임 정보 (에코 강도/이동, 카운트다운 산정) |
| 강수량 시간자료 | 기상청 AWS/ASOS | 급류 도달 시각의 근거, 결과 화면 대조 |
| 기상특보 발표 이력 | 기상청 API허브 / 재난안전데이터공유플랫폼 | 주의보/경보 학습 요소, 타임라인 |
| 긴급재난문자 발송 이력 | 공공데이터포털 (행정안전부) | 결과 화면의 반전: 문자 도착 시각 vs 레이더가 위험을 보여준 시각 |
| 자연재난 국민행동요령 | 공공데이터포털 (행정안전부) | 태스크 우선순위 채점의 공식 근거 |
| 대피소 위치 | 공공데이터포털 (행정안전부) | 엔딩 우리 동네 모드 |

## 5.3 사례 후보

2026년 7월 초 충청권 집중호우(청주, 천안 일대 침수)를 1순위 후보로 검토한다. 현장 해커톤 개최지(천안)의 최근 실측 데이터라는 점에서 시연 임팩트가 크다. 계곡 급류 시나리오에 더 적합한 사례가 있으면 교체하되, 레이더 시계열 확보 가능 여부를 먼저 검증한다.

## 5.4 API 키 보안

수집 스크립트는 로컬에서만 키를 사용하고 결과 정적 파일만 커밋한다. 런타임 API 호출은 없다 (참가자 가이드의 키 노출 실격 규정을 데모가 모범으로 시연).

# 8. 운영 리스크와 대응

1. 본선팀 아이디어와의 중복: 재난 시뮬레이션 계열 지원작이 존재한다. 데모는 구조(rescue)나 도시 방재 예산 배분이 아니라 "개인의 데이터 판독과 대피 타이밍"으로 범위를 명확히 구분하고, 문서에 "본선팀 제출물과 무관하게 운영진이 방법론 시연용으로 제작"을 명시하며, 공개 시점은 사전교육일에 전체 팀 동시 공개로 통일한다.
2. 안전 교육으로의 오독: 행동요령 암기가 아니라 기상 데이터 판독이 채점 대상임을 화면과 문서에서 반복해 명시한다 (주제 적합성 방어).
3. 데이터 미확보: 사례 후보의 레이더 시계열 확보를 다른 작업보다 먼저 검증한다. 확보 실패 시 사례를 교체하되 가상 데이터로 대체하지 않는다.
