# 예보관 의사결정 체험 (forecaster) 개발계획서

기상·기후 AI 해커톤 2026 서류 심사용 개발계획서(최종). 산출물은 웹 인터랙티브 학습 도구 forecaster — 학습자가 실제 기상청 데이터로 직접 날씨를 예측하고 실황과 비교하며, 앱이 손을 잡아주는 가이드로 예보의 원리와 확률적 사고를 익힌다.

이 문서는 forecaster-app-plan.md(v1.0 빌드 스펙)와 plan-revision-v1.1(가이드 우선 개정)을 통합하고, climate-edu 시제품 평가에서 얻은 작동·배포 교훈을 반영한 것이다. 각 장은 공고 심사 5개 항목의 세부 요소에 1:1로 대응한다.

## 0. 한 줄 요약

중3~고1과 일반 시민(전공지식 0)이, 실제 기상청 데이터를 보고 직접 "비가 올까?"를 판단한 뒤 실황과 비교한다. 앱이 단계마다 다음 할 일을 안내해 누구나 끝까지 완주하며 확률적 사고를 체험으로 이해한다.

## 1. 주제 적합성 (배점 20)

1.1 무엇을 가르치는가: 예보관의 의사결정 과정 — 위성·레이더·수치모델·지상관측을 종합해 판단하고, 실황으로 검증하는 한 사이클.

1.2 특정 기상·기후 원리: 데이터 종류별 의미(위성=구름, 레이더=현재 강수, 모델=예측, 관측=실측), 모델 불일치, 확률예보, 실황 검증.

1.3 대상·단원 연결(현장 적용성): 주 대상 중3 과학 '기권과 날씨', 고1 통합과학 연계, 일반 시민도 가능. 한 차시(45~50분)에 1~3개 케이스.

## 2. 체험·참여형 설계 (배점 25, 1순위)

5단계 능동 의사결정 루프 + 가이드 시스템(v1.1).

| 단계 | 학습자 조작 | 결과(반응) |
|---|---|---|
| ① 브리핑 | 케이스 진입 | 상황·임무 제시 |
| ② 데이터 탐색 | 위성·레이더·모델·관측 레이어 토글 + 시간 슬라이더 | 선택 시각의 영상·수치 표시, 모델 불일치 강조 |
| ③ 예측 입력 | 확신 게이지 슬라이더 + 근거 체크 | 확신의 무게가 색·표정으로 즉시 반영 |
| ④ 실황 공개 | 제출 | 정답 공개 + 서사형 피드백 |
| ⑤ 리플렉션 | "무엇을 놓쳤나" 입력 | 맞춤 피드백, 대시보드 누적 |

2.1 양방향성: 토글·슬라이더·라디오·자유입력 전부 입력 → 반응 구조.

2.2 완결성: 튜토리얼 Case 0(점수 없는 따라하기) → easy → seoul(모델 불일치) → trap(심화 잠금)으로 시작–진행–완료가 닫힌다.

2.3 현장 적용성: 무설치 URL, 한 차시 분량, 자기완결(외부 교육자료 0개), 모바일 대응.

2.4 가이드 우선 시스템(v1.1, PhET보다 쉽게): 빈 화면에 던지지 않는다.

- 다음 할 일 배너: "레이더를 보고 비구름 위치를 찾아보세요" 상단 1줄 고정.
- 코치마크: 튜토리얼·첫 케이스에서 클릭할 곳을 스포트라이트로 순서 안내.
- 인라인 용어칩: 위성·레이더·모델·관측에 1줄 설명 상시.
- 막힘 방지 힌트: 무동작 시 다음 행동 노출. 키보드·스크린리더·reduced-motion 대응.

## 3. 학습 효과 (배점 10)

3.1 학습 목표(관찰 가능한 동사): 학생은 ① 위성·레이더·모델·관측이 각각 무엇을 알려주는지 구분한다 ② 강수확률을 '확신의 정도'로 해석한다 ③ 모델이 틀릴 수 있음을 사례로 설명한다.

3.2 대상·난이도 적합성: v1.1로 진입장벽을 낮춤 — easy→medium 우선 공개, trap은 심화 잠금. 화면엔 숫자 대신 쉬운 말.

3.3 측정 구조(이 도구의 강점): 내부 Brier 채점(강수확률+형태+기온)으로 정량 측정 + 성장 대시보드가 케이스별 정확도 추세를 가시화. 화면엔 서사 피드백, '왜 이 점수?' 펼침에서만 정량.

3.4 평가 기준: 케이스 완주 + 정확도 추세 향상 + 리플렉션 작성 여부로 "학습됨"을 판단.

## 4. 아이디어 독창성 (배점 15)

4.1 차별점(비교 대상 호명): windy·earth.nullschool은 현상 시각화, PhET는 단일 개념, MetEd는 강의형이다. forecaster는 "직접 예보 → 실황 검증 → 확률적 사고"를 실데이터로 체험하게 한다.

4.2 새로운 접근(v1.1 신규): 확률을 숫자(Brier) 없이 '확신 게이지 + 서사 피드백'으로 체감시킨다 — 통계 개념을 전공지식 0 학습자에게 전달하는 새 방식.

## 5. 실현 가능성 (배점 30) — 본선 작동 보장 포함

5.1 (a) 개발 단계·기능: Astro 정적 출력 + island 인터랙션. 1케이스 완전체를 완성하고 나머지는 동일 스키마 빈 슬롯으로 확장 가능성만 증명.

5.2 (b) 기상 데이터 1종 이상 명시·활용: 기상청 API허브(위성 gk2a, 레이더 HSR, 지상 sfctm) + 공공데이터포털 단기예보(getVilageFcst, 레포에 검증된 forecast.js 존재)·ASOS 실황. 모델 불일치는 동일 날짜를 두 발표시각(0200/0800)으로 호출해 실데이터로 재현. 원본 JSON을 raw/에 보존해 증빙.

5.3 (c) 1개월 내 현실성: v1.1 변경은 표시 레이어 중심이라 리스크가 낮고, 핵심 데이터·채점 엔진은 그대로 유지.

5.4 (d) AI 도구 활용: 리플렉션 맞춤 피드백(Claude, 서버리스), 브리핑 삽화·예보관 캐릭터 생성. 판단 근거 데이터는 100% 실측으로 유지하고 실데이터·AI 생성물을 화면 라벨로 구분.

5.5 (e) 구동 환경·배포 보장(climate-edu 평가 교훈 반영): 학교 환경에서 즉시 시연되도록 아래를 확정.

- 단일 앱 디렉토리로 구성(루트와 소스에 앱 폴더가 중복돼 라우팅이 깨지는 사고 금지 — climate-edu가 그 이유로 전체 404였음).
- 정적 출력 + 사전 수집 데이터 → 현장 네트워크 장애와 무관하게 오프라인 동작.
- API 키는 개발 시 수집 스크립트에서만 사용, 런타임·클라이언트 노출 0(실시간 호출 의존 제거).
- Vercel 배포 URL 라이브 접속 + 모바일 반응형을 제출 전 현장 점검.

## 7. climate-edu 시제품 평가에서 얻은 교훈 반영

| 교훈 | 영향 항목 | forecaster 반영 |
|---|---|---|
| 루트/소스 앱 폴더 충돌로 전체 404 | 실현 가능성 | 단일 앱 디렉토리 + 배포 라이브 검증 |
| 실시간 데이터가 키 없으면 죽음 | 실현 가능성 | 정적 사전 수집으로 런타임 키 의존 원천 제거 |
| 대상 학년·단원 불명 | 주제·학습 효과 | 중3 '기권과 날씨' 단원·목표 명시 |
| 측정 구조 약함 | 학습 효과 | Brier 채점 + 성장 대시보드 명문화 |
| 실측/시뮬 구분 모호 | 실현 가능성·부정행위 | 실데이터·AI 생성물 화면 라벨 구분 |

## 8. 작업 순서 (v1.1 §8 기준 + 배포 검증)

1. 단일 앱 디렉토리로 정리 + 로컬·Vercel 빌드 무오류 확인
2. lib/feedback.ts(점수→서사) + scoring.ts(내부 Brier) 구현·테스트
3. 가이드 시스템(NextActionBar → Coachmark → TermChip)
4. 튜토리얼 Case 0(데이터·스크립트·자동 진행)
5. easy 케이스 + seoul 완전체 데이터 수집(원본 JSON 보존)
6. 확신 게이지·서사 피드백 결합, 대시보드는 점수 있는 케이스만 추세
7. 중3 단원·학습 목표를 온보딩 화면과 문서에 표기
8. 배포 URL 라이브·모바일 현장 점검
