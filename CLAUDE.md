CLAUDE.md — 기상·기후 AI 해커톤 2026 콘텐츠 개발 가이드
이 문서의 목적
이 파일은 Claude Code가 이 저장소(대표 데모 「골든타임: 물이 차오르기 전에」)에서 코드를 작성하거나 수정할 때 우선적으로 따라야 하는 규칙을 담고 있습니다. 개발을 시작하기 전 `docs/PLAN.md`와 함께 확인합니다.
0. 개발 시작 전 확인사항

* `docs/PLAN.md`를 먼저 읽고, 그 안에 정의된 기획 범위와 기능 명세를 벗어나지 않는 범위에서 작업한다.
* `docs/PLAN.md`와 실제 구현이 달라지는 경우, 임의로 진행하지 말고 `docs/PLAN.md`를 먼저 갱신한 뒤 개발한다.
* 코드를 작성하기 전, 아래 1~4항의 원칙을 위반하지 않는지 스스로 점검한다.

1. 학습 대상 및 학습 목표

* 콘텐츠를 설계하기 전, 대상 학습자(연령대·학년·배경지식 수준)를 한 문장 이상으로 구체적으로 정의하고 README에 명시한다.
   * 예: "초등 고학년(4~6학년)을 대상으로, 기압과 날씨 변화의 관계를 이해시킨다."
* 학습 목표는 "기후를 이해한다"와 같은 추상적 표현이 아니라, 학습자가 콘텐츠를 마친 뒤 할 수 있게 되는 구체적 행동으로 서술한다.
   * 예: "기온·습도 데이터를 보고 강수 확률을 예측할 수 있다."
* 게임 요소(재미)와 학습 목표(개념 습득)는 동일한 플레이어 행동에서 동시에 발생하도록 설계한다. 미니게임을 클리어한 뒤 지식이 별도 텍스트로만 제공되는 것처럼 재미와 학습이 분리된 구조는 지양한다.
* 기상청 실데이터(API)는 콘텐츠를 구동하는 엔진으로 활용하되, API 사용 자체를 학습자에게 과시하는 UI 요소로 노출하지 않는다.

2. 독창성 확보 및 표절·중복 방지

* 기존에 공개된 교육 콘텐츠·웹사이트·앱의 UI, 텍스트, 시나리오, 이미지 등을 그대로 복제하지 않는다. 참고한 콘텐츠가 있다면 출처를 남기고 구조와 표현을 독자적으로 재구성한다.
* 생성형 AI(Claude 등)로 코드·텍스트·이미지를 생성한 경우에도, 결과물이 특정 원본 저작물을 그대로 재현하지 않았는지 확인한다.
* 팀 간 아이디어 중복은 표절과 별개로 확인 대상이 될 수 있다. 기획 초기 단계에 겹치는 주제·컨셉이 있다고 판단되면 임의로 진행하지 말고 운영진(BCuniverse/KPC)에 문의한다.
* Claude와의 대화 세션 링크는 제출 시 진위 증빙 자료로 쓰이므로, 팀 대표 계정으로 개발 세션을 일원화하고 세션을 임의로 삭제하지 않는다.

3. 라이선스 및 저작권 관리

* 프로젝트에서 사용하는 모든 외부 리소스(이미지, 폰트, 아이콘, 사운드, 오픈소스 라이브러리, 데이터셋, 템플릿 등)는 사용 시점에 `LICENSES.md`(또는 README 내 별도 섹션)에 아래 정보를 기록한다.
   * 리소스명 / 출처(URL) / 라이선스 종류(MIT, CC-BY 4.0 등) / 상업적 이용 가능 여부 / 변형(수정) 여부
* 라이선스가 불분명하거나 출처가 확인되지 않는 리소스는 사용하지 않는다. 저작자 표시(CC-BY 등)가 요구되는 리소스는 화면 내 크레딧 또는 README에 반드시 표시한다.
* 기상청 데이터 등 공공데이터 사용 시, 출처와 이용 조건(공공누리 유형 등)을 명시한다.
* 새 외부 라이브러리를 추가할 때는 그 라이선스가 프로젝트 배포 형태(웹 서비스)와 충돌하지 않는지 확인한다. GPL 계열 라이브러리는 특히 주의한다.
* 제출 시 별도로 요구되는 "AI 활용 내역서", "외부자원 사용 명세서", "라이선스 명세서" 작성을 염두에 두고, 개발 중 사용한 리소스와 AI 프롬프트 활용 내역을 그때그때 기록해 둔다.

4. 보안 — API 키 및 민감정보 유출 방지

* API 키, 토큰, 비밀번호 등 민감정보는 코드에 직접 하드코딩하지 않는다. 반드시 `.env` 파일에 저장하고 환경변수로 불러온다.
* `.env`(및 이에 준하는 시크릿 파일)는 `.gitignore`에 반드시 포함시켜 Git 저장소에 커밋되지 않도록 한다. 저장소 초기화 시 `.gitignore`부터 확인·생성한다.
* `.env.example` 파일을 별도로 만들어 필요한 환경변수 이름만 공유하고, 실제 값은 포함하지 않는다.
* 커밋 전에는 변경사항(diff)에 키·토큰 문자열이 포함되어 있지 않은지 항상 확인한다. 실수로 키가 커밋된 이력이 있다면 즉시 해당 키를 재발급(rotate)하고 운영진에 알린다.
* 브라우저에 노출되면 안 되는 키는 서버리스 함수·백엔드를 경유하도록 구조를 설계한다. 클라이언트 사이드 코드에 비밀 키를 직접 심지 않는다.
* 배포(Vercel 등) 시에는 플랫폼의 환경변수 설정 기능을 통해 값을 등록하고, 코드에 값이 남지 않도록 한다.

5. README 작성 규칙
이 저장소 루트에는 아래 항목을 포함한 README.md를 작성한다.

1. 프로젝트 개요: 콘텐츠명, 한 줄 소개, 대상 학습자, 학습 목표
2. 실행 방법: 로컬 개발 환경에서 실행하기 위한 단계별 명령어(설치, 환경변수 설정, 실행)
3. 배포 방법: 실제 구동 URL과, 동일하게 재배포하기 위한 절차(배포 플랫폼, 빌드 명령어 등)
4. 외부 리소스 출처 및 라이선스: 3항에서 기록한 리소스 목록 요약 또는 `LICENSES.md` 링크
5. 사용한 기상 데이터 및 API: 어떤 데이터를 어떤 API로, 어떤 목적으로 사용했는지
6. 팀 정보: 팀명 및 구성원 역할(개인정보 과다 노출은 지양)

6. Claude Code 작업 시 유의사항

* 이 CLAUDE.md에 명시된 규칙을 코드 생성·수정 제안 시 항상 우선 적용한다.
* 외부 리소스를 추가하는 코드를 작성할 때는 해당 리소스의 출처·라이선스를 `LICENSES.md`에 함께 기록한다.
* API 키가 필요한 기능을 구현할 때는 `.env` 사용 여부와 `.gitignore` 반영 여부를 함께 확인한다.
* 커밋 메시지에 시크릿 값이 포함되지 않도록 주의한다.
