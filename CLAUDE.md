# 프로젝트 개요

기상·기후 AI 해커톤 2026 출품작. 대표 데모는 「골든타임: 물이 차오르기 전에」 —
집중호우·도시 침수 상황에서 대피 시간을 계산하고 안전한 경로를 판단하는 웹 기반
교육 게임(중학생 대상)이다.

**`docs/PLAN.md`(개발계획서, 공고 양식2 대응)를 항상 우선 참고한다.** 개발 목적·
대상·활용 데이터·학습 목표·평가 설계·개발 일정이 이 문서에 확정되어 있다 —
코드/콘텐츠가 이 계획과 어긋나면 계획을 먼저 갱신하고 작업한다.

> 레포에는 이전 컨셉인 「예보관 의사결정 체험」(`forecaster-experience/`, 관련
> 문서 `docs/forecaster-app-plan.md`·`docs/forecaster-build-spec.md`·
> `docs/forecaster-data-utilization.md`·`docs/forecaster-ux-spec.md`)의
> 스캐폴드도 남아 있다. 신규 작업은 골든타임(`docs/PLAN.md`) 기준으로 진행하고,
> forecaster-experience 문서 내용을 참고할 때는 `docs/PLAN.md`와 상충하지
> 않는지 먼저 확인한다.

## 학습 대상 및 목표

콘텐츠를 추가·수정할 때는 아래 학습 대상·목표에 부합하는지 확인한다.

- **대상**: 중학생(교육과정 연계: 2015 개정 중3 과학 「기권과 날씨」·「재해·재난과
  안전」, 2022 개정 [9과17-04]). 재난 상황에서 시간·경로를 스스로 계산하고
  판단해본 적이 없는 학습자를 우선한다.
- **학습 목표**(`docs/PLAN.md` §6.1):
  1. 이동 거리·이동 수단·경로 위험도를 고려해 대피 소요 시간을 계산할 수 있다.
  2. 강수강도가 만든 골든타임과 자신의 대피 시간을 비교해 안전한 경로를 판단할
     수 있다.
  3. "가장 가까운 길이 항상 안전하지 않다"는 원칙을 근거를 들어 설명할 수 있다.
  4. 호우 시 피해야 할 위험 요소(침수 도로, 지하공간, 하천변, 맨홀, 감전 위험
     시설)를 식별할 수 있다.
- 새 요소를 추가할 때 위 학습 목표와 무관한 장식적 보상(XP·배지 등)을 넣지 않는다
  — 재미 요소(물 차오름 게이지, 카운트다운, 위험 경고 등) 자체가 학습 개념이
  되도록 설계한다(`docs/PLAN.md` §7.1).

## 표절 및 아이디어 중복 방지

- 새 케이스·UI 패턴·비주얼 컨셉을 추가하기 전, 국내외 유사 기상 교육 콘텐츠(기상청
  교육 자료, 타 해커톤 출품작 등) 및 이번 해커톤 타 팀의 공개 자료와 겹치지 않는지
  확인한다. 확인 근거(검색어·비교 대상·결론)를 PR 설명 또는 `docs/`에 남긴다.
- 유사한 아이디어가 이미 존재한다면 그대로 재현하지 말고 차별점(데이터 활용 방식,
  케이스 구성, 인터랙션)을 명시적으로 설계한다.

## 라이선스 및 저작권

- 외부 리소스(폰트, 라이브러리, 이미지·영상 소재, 데이터셋)를 추가할 때는 라이선스
  종류와 교육·해커톤 용도 사용 허용 여부를 먼저 확인한 뒤에만 반영한다.
- 각 리소스의 출처·라이선스는 사용처의 `README.md`에 기록한다 — 리소스명, 출처
  URL, 라이선스명(예: MIT, CC-BY-4.0), 수정 여부.
- AI로 생성한 이미지·문구는 "AI 생성물"임을 표기하고, 실측 수치 데이터(위성·레이더·
  ASOS·예보·실황)와 혼동되지 않도록 화면에 출처 라벨을 구분한다.
- 기상청 API허브·공공데이터포털 데이터는 각 서비스의 이용약관·출처표기 의무를
  준수하고, 수집 스크립트와 원본 raw 데이터를 보존해 검증 가능하게 한다.
- 저작권이 불명확한 소재는 사용하지 않는다. 확인이 끝나기 전까지 PR에 포함하지 않는다.

## 보안 — API 키 유출 방지

- `.env*` 파일은 커밋하지 않는다(레포 전역 `.gitignore`에 이미 등록됨). 새 하위
  프로젝트를 추가할 때도 동일하게 적용한다.
- API 키(`KMA_API_KEY`, `KMA_HUB_AUTHKEY`, `ANTHROPIC_API_KEY` 등)는 서버리스
  함수(`api/*.js`)에서만 참조한다. 클라이언트 번들·정적 파일·콘솔 로그에 키가
  노출되지 않는지 커밋 전 확인한다.
- 데이터 수집 스크립트(예: `forecaster-experience/scripts/`)는 로컬에서만 키를
  사용하고, 커밋되는 것은 수집 결과(정적 JSON/이미지)뿐이어야 한다.
- 새 `.env.example`을 추가할 때는 키 이름만 적고 실제 값은 절대 채우지 않는다.
- 커밋하기 전 `git diff`/`git status`로 우발적인 키·시크릿 포함 여부를 확인한다.

## README 작성 규칙

각 하위 프로젝트(`forecaster-experience/` 등)의 `README.md`에는 최소한 아래 항목을
포함한다.

1. **실행 방법** — 로컬 개발 서버 실행 커맨드(`npm install`, `npm run dev` 등)
2. **배포 방법** — 배포 대상(Vercel 등)과 필요한 환경변수/설정
3. **외부 리소스 출처 및 라이선스** — 사용한 폰트·라이브러리·이미지·데이터셋의
   출처와 라이선스를 표로 정리(예: 리소스명 | 출처 | 라이선스 | 비고)

README 갱신 없이 새 외부 리소스를 추가하는 PR은 병합하지 않는다.
